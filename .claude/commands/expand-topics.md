Пополни `drafts/_topic-map/$ARGUMENTS.json` десятью новыми темами для категории `$ARGUMENTS`. Запускаешься scheduler-ом, когда свободных (неиспользованных) тем в этом файле не осталось.

## Что НЕЛЬЗЯ

- Перезаписывать или удалять существующие темы в файле. Только добавлять в массив `topics`.
- Дублировать темы, которые уже есть в файле (любой `slug` или `main_keyword`).
- Дублировать slug-и, которые есть в `data/published_index.json` (это уже опубликованные статьи).
- Дублировать slug-и, которые есть в подпапках `drafts/{slug}/` (это статьи в работе).
- Писать саму статью. Только бриф темы.

## Heartbeat (обязательно)

В начале работы и перед каждым WebSearch обнови heartbeat-файл:

```bash
date -u +"%Y-%m-%dT%H:%M:%S | expand-topics" > data/.scheduler_heartbeat
```

Без этого scheduler убьёт subprocess через 5 минут «тишины».

## Алгоритм

### 1. Прочитай текущее состояние

- `drafts/_topic-map/$ARGUMENTS.json` — текущий список тем категории.
- `data/published_index.json` (поле `articles[].slug` и `articles[].main_keyword`) — уже опубликованные.
- Список папок в `drafts/` — статьи в работе. Слагы которые НЕ начинаются с `_`.

Собери единый набор «занятых slug-ов» и «занятых main_keyword» — новые темы должны не пересекаться ни с одним из них.

### 2. Прочитай стайл-гайды (источник правды)

- `.claude/style/yandex-quality.md` — правила Баден-Баден (плотность ключа, естественность).
- Если `category=news` — также `.claude/style/news-sourcing.md` (шесть тематических зон, окно свежести 30 дней).
- `.claude/agents/1-semantics.md` — формат brief-полей, которые ждут следующие агенты.

### 3. Подбери 10 новых тем

**Источники идей (в порядке приоритета):**

1. **Свежая судебная практика 2024-2026 годов** — определения ВС РФ, постановления Пленума, обзоры судебной практики. WebSearch по `vsrf.ru`, `pravo.gov.ru`, `garant.ru`, `consultant.ru`, `klerk.ru`.
2. **Изменения в законе 127-ФЗ «О банкротстве»** за последние 2 года.
3. **Поисковые подсказки** — что реально гуглят люди по теме категории. WebSearch по типичным запросам категории, посмотри связанные запросы и «люди ищут также».
4. **Соседние подкатегории**, которые ещё не покрыты в текущем topic-map. Проанализируй existing темы, найди пропуски (например, есть «банкротство пенсионера», нет «банкротство ИП» — добавь).
5. **Сезонные/событийные темы** — конец налогового года, мораторий, индексация прожиточного минимума, новые лимиты.

**Распределение по интентам и стадиям воронки** (важно для конверсии в лиды, не делай все 10 тем одного типа):

- 3-4 темы с интентом `commercial` или `problem-aware` на стадии `decision` — основной источник лидов.
- 3-4 темы `solution-aware` на стадии `consideration` — закрытие возражений.
- 2-3 темы `informational` на стадии `awareness` — для прогрева трафика и охвата.

### 4. Особое поведение для category=news

Только новости за последние 30 дней. Применяй раздел «Алгоритм агента 1 при category=news» из `.claude/style/news-sourcing.md`. Каждая тема должна быть из одной из шести зон (`vs_practice`, `vs_plenum`, `legislation`, `moratorium`, `initiative`, `finance`) и иметь `event_date`, `news_zone`, `primary_source`.

**ХАРД-ПРАВИЛО (с 12 мая 2026):** для category=news поля `event_date` (ISO YYYY-MM-DD), `news_zone` (одно из 6 значений выше), `primary_source` (валидный URL, реально проверенный WebFetch-ом) **ОБЯЗАТЕЛЬНЫ**. Тема без хотя бы одного из этих полей **гарантированно будет помечена rejected** scheduler-функцией `_sanitize_news_topics` сразу после твоего завершения. Если ты не можешь найти конкретное событие с датой и URL источника — НЕ добавляй тему. Лучше вернуть меньше тем (минимум 3 реально валидных), чем дописать мусор для счёта.

Если за прогон не нашлось 10 свежих новостей — допиши столько, сколько есть (минимум 3). Не выдумывай старые события. Не пиши обобщённые «темы про X» без привязки к конкретному событию.

**ЖЁСТКОЕ правило для category=news: год в title и h1 запрещён.** Никаких исключений.

Запрещены ВСЕ варианты 4-значного года в `title` и `h1`:
- Голый год: `2024`, `2025`, `2026`, `2027`.
- В сочетании: «в 2026 году», «с 2025 года», «практика 2026», «за 2025-2026».
- Номер документа с годом: «обзор №5/2026», «№40/2024».
- Дата подписания/принятия в названии: «ФЗ-259 от 08.08.2024», «постановление №40 от 17.12.2024».

Документы упоминай только без года в `title` и `h1`: «обзор Верховного суда», «ФЗ-259», «постановление Пленума №40». Год и точные даты можешь (и должна) использовать в `description`, `rationale` и в `event_date` — там они раскрывают контекст. Вместо «Мораторий на банкротство в 2026 году» пиши «Мораторий на банкротство: что осталось от запретов». Вместо «Обзор ВС РФ №5/2026 по банкротству» — «Новый обзор Верховного суда по банкротству».

Заказчик хочет, чтобы статья news оставалась релевантной в поиске и через год после публикации. Год в `title` мгновенно делает её «старой».

**Хард-проверка**: после агента 6 скрипт `articles_scheduler/finalize_draft.py` валит слот в `failed_qa`, если для `category=news` в `title` или `h1` найдёт `\b(19|20)\d{2}\b`. То же самое произойдёт, если ты сюда положишь тему с годом — следующий слот по этой теме упадёт.

### 5. Сформируй структуру каждой темы

Каждая новая тема в массиве `topics` должна содержать ровно эти поля (формат как у существующих в файле, см. первую тему как образец):

```json
{
  "id": "fiz-09",
  "slug": "kratkij-slug-temy-2026",
  "title": "Заголовок темы (50-60 знаков)",
  "title_length": 56,
  "description": "Meta description темы, 130-160 знаков.",
  "description_length": 145,
  "h1": "H1 для статьи (может отличаться от title, 50-70 знаков)",
  "main_keyword": "главный поисковый запрос темы",
  "main_keyword_density_target": "≤2.5%",
  "secondary_keywords": [
    "вторичный ключ 1",
    "вторичный ключ 2",
    "вторичный ключ 3",
    "вторичный ключ 4",
    "вторичный ключ 5",
    "вторичный ключ 6"
  ],
  "intent": "commercial | problem-aware | solution-aware | informational",
  "funnel_stage": "awareness | consideration | decision",
  "article_type": "step-by-step | comparison | case-study | law-explanation | faq | myths",
  "offer": "proverka-spisaniya | otmena-prikaza | snyatie-aresta | raschet-stoimosti",
  "frequency_estimate": "high | medium | low",
  "rationale": "Зачем эту тему писать (1-2 предложения): какой запрос пользователя закрывает, почему сейчас актуально, как ведёт к лиду.",
  "expected_length_chars": 7000,
  "topic_action": "короткое описание действия для CTA, например 'банкротство при ипотеке'"
}
```

**ВАЖНО про поле `status`:** в новых темах НЕ указывай его. Scheduler пропускает только темы со `status='rejected'` (явный отказ заказчика), всё остальное — включая темы вообще без поля status — берёт в работу. Существующие темы со `status='proposed'` оставь как есть, не редактируй.

**Нумерация id:** найди максимальный номер среди существующих id вида `{category}-NN`, продолжай с него +1. Например, если последний `fiz-08` — новые `fiz-09`, `fiz-10`, ..., `fiz-18`.

**Slug-генерация:** транслитерация title в kebab-case на латинице, без стоп-слов («как», «что», «для», «при»). Длина 30-60 знаков. Должен быть уникальным в пределах ВСЕГО проекта.

### 6. Запиши результат

Прочитай оригинальный JSON файл, добавь 10 новых тем в массив `topics` (в конец, не вставляй между существующими), запиши обратно с тем же форматированием (UTF-8, 2 пробела отступ, без BOM).

```python
# Пример записи через Python (можно через Write/Edit tool):
import json
path = "drafts/_topic-map/fiz.json"
data = json.load(open(path, encoding="utf-8"))
data["topics"].extend(new_topics)  # new_topics = [10 объектов]
json.dump(data, open(path, "w", encoding="utf-8"), ensure_ascii=False, indent=2)
```

Не забудь финальный `\n` в конце файла.

### 7. Финальный отчёт

Выведи кратко:

- Сколько тем добавлено (`added=10`).
- Перечисли id и main_keyword каждой добавленной темы.
- Сколько тем в файле теперь всего (`total=18`).
- Если category=news: какие event_date / news_zone охвачены.

После твоего завершения scheduler сам закоммитит обновлённый `drafts/_topic-map/$ARGUMENTS.json` в репозиторий.

## Если не получилось

- Не нашёл WebSearch-ом ничего нового → допиши столько тем, сколько смог придумать на базе существующих пробелов в topic-map (минимум 3).
- Файл `drafts/_topic-map/$ARGUMENTS.json` не существует → создай его с базовой структурой `{"category": "$ARGUMENTS", "category_label": "...", "generated_at": "YYYY-MM-DD", "topics": [...]}`.
- Любая ошибка чтения/записи → не пытайся «починить» руками. Сообщи короткий текст ошибки и остановись с ненулевым exit (через `print()` ошибки и завершение).
